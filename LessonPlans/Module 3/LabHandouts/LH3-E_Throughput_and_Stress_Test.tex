\labheader{3-E}{Throughput Characterization and Multi-Station + PAUSE Stress Test}

\section{Objective}
Determine the system's throughput ceiling and \FIFO{} behavior under cooperative-target ping-pong with optional PAUSE-burst load, identify the pipeline's performance bottleneck, and generate the production bitstream (without \mbox{ILA}) that will be used for all data collection in Lab Handouts~3-F through 3-H.

\section{Prerequisites}
\begin{itemize}[leftmargin=2em]
	\item Lab Handout~3-D complete (\PER{} and latency characterized for a single station).
	\item Target~A, Target~B, and the Threat all powered and configured with known Beacon~\mbox{ID}s and transmission schedules.
\end{itemize}

\section{Background: Throughput Analysis}

The theoretical maximum frame rate is constrained by the \UART{} link between the STM32 interceptor (Threat) and the \FPGA{}:
\[
	\text{Throughput}_{\text{max}} = \frac{\text{Baud Rate}}{\text{Bits per Frame}} = \frac{115{,}200}{43 \times 10} = 268~\text{fps}
\]
In practice, the \RF{} on-air time imposes a much lower ceiling for cooperative-target traffic. At SF7 with 1-second ping-pong (DATA + ACK pairs), the expected frame rate is about 2~fps --- far below the \UART{} ceiling. PAUSE bursts at 10\,Hz from the Threat add up to 10 fps when the button is held, but most are half-duplex shadowed and never make it to the \FPGA{}. The stress test is therefore about finding the realistic operating envelope and verifying the \FIFO{} does not overflow when intercepted-packet bursts cluster.

\section{Procedure}

\subsection{Step 1 --- FIFO utilization visibility}
\begin{enumerate}[leftmargin=2em]
	\item Map the 6-bit \texttt{fifo\_count} signal to \texttt{led(5:0)} on the Nexys A7 (a simple binary display). This provides real-time \FIFO{} occupancy visible without the \mbox{ILA}.
	\item Map \texttt{fifo\_full} (or the \FIFO{} overflow flag from LH3-D) to \texttt{led(15)} as a latching alarm: once set on overflow, it stays set until reset is pressed.
	\item Re-synthesize and re-implement. This does not require a full debug bitstream; use the \texttt{g\_DEBUG = true} version for now (you will generate the production bitstream in Step~8).
\end{enumerate}

\begin{tcolorbox}[infobox={Why Map FIFO to LEDs?}]
	The \mbox{ILA{}} requires a JTAG connection and Vivado running on the host PC. During a 5-minute stress test it is impractical to watch the \mbox{ILA} continuously. The \mbox{LED} display provides a zero-overhead real-time indicator: you can see at a glance whether the \FIFO{} is filling up, and the overflow alarm \mbox{LED} tells you if you missed a burst at any point during the run.
\end{tcolorbox}

\subsection{Step 2 --- Single-station baseline}
\begin{enumerate}[leftmargin=2em]
	\item Power Target~A only: SF7, +14~dBm, 2-second fixed initiator interval. Target~B off, Threat button not held. Only DATA rows from Beacon~ID \texttt{0x01} are produced.
	\item Run for 5 minutes. Record: packets transmitted ($N_{\text{tx}}$), \FPGA{} valid frames ($N_{\text{valid}}$), \PER{}, peak \FIFO{} occupancy (\texttt{led(5:0)} peak), overflow alarm status.
	\item This is the performance baseline row in the table below.
\end{enumerate}

\subsection{Step 3 --- Cooperative-target ping-pong (fixed initiator)}
\begin{enumerate}[leftmargin=2em]
	\item Power on Target~B (responder). Target~A initiates DATA every 1\,s; Target~B responds with ACK on each successful DATA.
	\item Run for 5 minutes. Record metrics for both Beacon~IDs (sum $N_{\text{tx}}$ across DATA + ACK rows; $N_{\text{tx}}^{\text{DATA}}$ from Target~A's USART2, $N_{\text{tx}}^{\text{ACK}}$ from Target~B's USART2).
	\item Expected frame rate: $\sim$2~fps (one DATA + one ACK per second). Expected \FIFO{} peak: low.
\end{enumerate}

\subsection{Step 4 --- Cooperative-target ping-pong (ALOHA initiator)}
\begin{enumerate}[leftmargin=2em]
	\item Reconfigure Target~A to ALOHA initiator mode (random delay uniformly distributed in [1,3]\,s). Target~B remains an event-driven ACK responder.
	\item Run for 5 minutes. Record \PER{} and watch for any increase in the \FIFO{} overflow alarm. ACK arrivals tightly follow each DATA, so the dominant burst pattern is DATA-then-ACK pairs separated by the SF7 round-trip ($\sim$60\,ms).
\end{enumerate}

\subsection{Step 5 --- Cooperative-target ping-pong + sustained PAUSE bursts}
\begin{enumerate}[leftmargin=2em]
	\item Restore Target~A to fixed 1\,s initiator interval (so the baseline traffic is deterministic).
	\item Have an operator hold the Threat's user button (B1) for the entire 5-minute run. The Threat will emit PAUSE packets at 100\,ms cadence; most are half-duplex shadowed by the Threat's own RX path, but a small fraction reach the \FPGA{} (visible as \texttt{pkt\_type = 10} rows from Beacon~ID \texttt{0xFF} in the \mbox{ILA{}}).
	\item During this run the targeted Target's DATA TX is gated by the 500\,ms hold-off, so Target~A's effective TX rate drops sharply; Target~B will have nothing to ACK during pauses. Record the resulting \PER{} measured against the originating sequence numbers --- expect high apparent ``\PER{}'' for the targeted Target because the Target genuinely did not transmit during the held window. Document this as expected behavior, not a fault.
	\item Watch \texttt{led(5:0)} during this run. The PAUSE-induced gating typically suppresses bursting, so \FIFO{} peak should remain low.
\end{enumerate}

\subsection{Step 6 --- Maximum contention (ALOHA + intermittent PAUSE bursts)}
\begin{enumerate}[leftmargin=2em]
	\item Configure both Targets for ALOHA initiator mode (Target~A in [0.5,2]\,s; Target~B initiating every [0.5,2]\,s in addition to acting as a responder, if your LH1-F firmware supports a dual-role responder build).
	\item Have the operator hold the Threat's user button intermittently --- 5\,s held, 5\,s released --- for the duration of the 5-minute run. This produces the most varied traffic mix in the dataset: ALOHA collisions, PAUSE bursts, and post-PAUSE recovery transients.
	\item This is the maximum contention configuration and the most likely to produce \FIFO{} overflow under burst conditions.
	\item If the overflow alarm (\texttt{led(15)}) sets during this run, note the \mbox{LED} state at the time and record when in the run it occurred.
\end{enumerate}

\subsection{Step 7 --- Bottleneck identification and mitigation proposal}
\begin{enumerate}[leftmargin=2em]
	\item Examine the stress test table. If \FIFO{} overflow occurred, identify the configuration that triggered it. The bottleneck is the link between the overflow source and the pipeline stage that cannot drain fast enough.
	\item For a \FIFO{}-dominated bottleneck, propose at least one mitigation: increase the \FIFO{} depth from 64 to 256 entries (uses one additional \mbox{BRAM} tile), upgrade the interceptor-to-\FPGA{} link from \UART{} to \mbox{SPI} (faster), or reduce the inter-frame overhead in the Threat firmware (shorter frame gaps).
	\item Document the bottleneck and proposed mitigation in one paragraph. You do not need to implement the mitigation for this handout.
\end{enumerate}

\begin{tcolorbox}[infobox={Why Not Just Increase the FIFO Depth?}]
	A deeper \FIFO{} delays overflow but does not eliminate the root cause. If the peak ingress rate (bytes/s from the \UART{}) permanently exceeds the egress rate (bytes/s the parser can process), no finite \FIFO{} depth will prevent eventual overflow. In practice, the Module~2 parser processes one byte per clock cycle at 100\,MHz, which is $\gg$115200 baud. The actual risk is transient bursts from multiple simultaneous receptions (DATA + ACK + a half-duplex-leaked PAUSE arriving in close succession), not a sustained throughput mismatch. Increasing the \FIFO{} depth from 64 to 256 entries is sufficient for the cooperative-target + PAUSE-burst configurations in this module.
\end{tcolorbox}

\subsection{Step 8 --- Generate the production bitstream}
\begin{enumerate}[leftmargin=2em]
	\item In \texttt{ingest\_top.vhd} (or the top-level instantiation), set \texttt{g\_DEBUG = false}. This removes the \mbox{ILA} core, freeing the 4 \mbox{BRAM{}} tiles it was occupying.
	\item Run Synthesis, Implementation, and Generate Bitstream. The output is \texttt{ingest\_top\_prod.bit}.
	\item Open the post-implementation utilization report. Confirm \mbox{BRAM{}} count is back to the Module~2 baseline (2 tiles for the feature \mbox{BRAM{}}, not 6).
	\item Program the production bitstream. The \mbox{ILA{}} dashboard will no longer be available, but all pipeline functionality and \mbox{LED{}} indicators (from Step~1) remain.
	\item Run the single-beacon PER test from Step~2 again with the production bitstream. Confirm the \PER{} is within 1 percentage point of the debug-bitstream measurement. Any large change indicates the \mbox{ILA{}} was interfering with timing in the debug build.
\end{enumerate}

\section{Verification Checklist}
\begin{itemize}[leftmargin=2em]
	\item[$\square$] \FIFO{} count mapped to \mbox{LEDs{}}; overflow alarm on \texttt{led(15)} latches on overflow.
	\item[$\square$] All five station configurations in the stress table completed and recorded.
	\item[$\square$] \FIFO{} peak utilization noted for each configuration; any overflow events documented.
	\item[$\square$] Bottleneck identified and mitigation proposed.
	\item[$\square$] Production bitstream generated (\texttt{g\_DEBUG = false}); \mbox{BRAM{}} count is 2 tiles.
	\item[$\square$] Production \PER{} within 1 percentage point of debug \PER{} (no timing regression).
\end{itemize}
