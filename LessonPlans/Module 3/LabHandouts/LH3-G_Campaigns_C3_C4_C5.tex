\labheader{3-G}{Data Collection --- Campaigns C3, C4, and C5}

\section{Objective}
Execute the three remaining single-variable sweep campaigns: modulation type (C3), physical distance (C4), and beacon count with both fixed-interval and \mbox{ALOHA} modes (C5). These campaigns complete the controlled experimental dataset that drives all machine learning work in Module~4.

\section{Prerequisites}
\begin{itemize}[leftmargin=2em]
	\item Lab Handout~3-F complete (collection script working; C1 and C2 CSV files validated).
	\item Tape measure available for Campaign~C4 distance measurements.
	\item All three beacons operational for Campaign~C5.
\end{itemize}

\section{Procedure}

\subsection{Campaign C3: Modulation Type Sweep}

\subsubsection{Setup}
Configure Beacon~A: +14~dBm, 3~m from receiver, 2-second fixed interval. Change only the modulation type: LoRa (spreading factor-based) and \FSK{} (frequency shift keying). Keep the receiver and \FPGA{} configuration unchanged --- the Module~2 parser was designed to handle both packet types, as the forwarding frame format is identical regardless of over-the-air modulation.

\begin{tcolorbox}[infobox={LoRa vs.\ FSK in this System}]
	The beacon's RFM95W (SX1276) supports both LoRa spread-spectrum and \FSK{} modulation. The feature vector fields (\mbox{RSSI}, \mbox{SNR}, spreading factor code, modulation code) differ between the two: \FSK{} does not use a spreading factor, so \texttt{spread\_factor} will report a sentinel value (0x00 in the LH1-F firmware). The modulation code field uses the canonical Module~1/2 mapping \textbf{\texttt{mod\_code} = 0x01 (LoRa), 0x02 (\FSK{}), 0x03 (\OOK{})}; it is the ground-truth label for the modulation classification task in Module~4. The feature distribution --- especially \mbox{RSSI}, \mbox{SNR}, and frequency error --- will differ between LoRa and \FSK{} because the SX1276 uses different receive paths and sensitivity levels for each modulation.
\end{tcolorbox}

\subsubsection{Execution}
\begin{enumerate}[leftmargin=2em]
	\item Run with LoRa modulation first: create \texttt{campaign\_c3\_lora.json} with \texttt{sweep\_variable: "mod"}, \texttt{sweep\_value: "lora"}, and \texttt{target\_packets: 300}. Collect 300 packets.
	\item Switch Beacon~A to \FSK{} mode (update firmware, reflash). Create \texttt{campaign\_c3\_fsk.json} and collect 300 packets.
\end{enumerate}

\subsubsection{Validation}
\begin{enumerate}[leftmargin=2em]
	\item Verify 600 total rows; 300 rows per modulation type.
	\item Confirm \texttt{mod\_code} = \texttt{0x01} for the LoRa run and \texttt{mod\_code} = \texttt{0x02} for the \FSK{} run (canonical Module~1/2 codes). Any mismatch is a firmware configuration error.
	\item Compare \mbox{RSSI} distributions between LoRa and \FSK{} at the same distance and power. \FSK{} typically produces slightly lower \mbox{SNR} than LoRa at the same \mbox{TX} power because LoRa's processing gain is not available. Plot histogram or violin plot of \mbox{RSSI} and \mbox{SNR} for both modulation types.
\end{enumerate}

\subsection{Campaign C4: Physical Distance Sweep}

\subsubsection{Setup}
Configure Beacon~A: +14~dBm, SF7, LoRa, 2-second fixed interval. Vary only the physical separation between the beacon and the receiver. Use a tape measure for all distance measurements and record to the nearest 0.5~m.

\begin{tcolorbox}[warning={Distance Measurement Protocol}]
	Measure distance from the \emph{antenna} of the beacon to the \emph{antenna} of the receiver, not board-to-board. In an indoor environment, multipath reflections from walls, furniture, and the ground plane are unavoidable; the measured \mbox{RSSI} will deviate from the free-space path loss model. Record the measurement method and any significant obstructions (e.g., ``metal shelving at 2~m between beacon and receiver'') in the campaign notes field. This information is required for the distribution shift analysis in LH3-H.
\end{tcolorbox}

\subsubsection{Execution}
\begin{enumerate}[leftmargin=2em]
	\item For each distance --- 1~m, 3~m, 5~m, 8~m --- place the beacon, confirm the receiver is receiving (check debug \UART{}), then run \texttt{collect.py} with the appropriate config JSON (\texttt{distance\_m} field must match). Collect 200 packets per distance.
	\item Between distance changes, verify the beacon is still transmitting (check debug \UART{}) and the receiver is still receiving before starting the next collection run.
\end{enumerate}

\subsubsection{Validation}
\begin{enumerate}[leftmargin=2em]
	\item Verify 800 total rows, 200 per distance.
	\item Compute mean \mbox{RSSI} per distance. It should decrease (become more negative) as distance increases.
	\item Overlay the theoretical free-space path loss (\mbox{FSPL}) curve. For the 915\,MHz carrier, TX power $P_{\text{tx}}$, and distance $d$:
	\[
		\text{\mbox{RSSI}}_{\text{theory}}(d) = P_{\text{tx}} + G_{\text{tx}} + G_{\text{rx}} - 20\log_{10}(d) - 20\log_{10}(915 \times 10^6) + 147.55
	\]
	where $G_{\text{tx}} = G_{\text{rx}} \approx 0$\,dBi for the spring antennas from LH1-A.
	\item Generate a scatter plot of measured mean \mbox{RSSI} vs.\ distance with the \mbox{FSPL} curve overlaid. Indoor measurements will typically show higher \mbox{RSSI} than \mbox{FSPL} at short distances (constructive multipath) and more variable behavior at longer distances.
\end{enumerate}

\labfigure{LH3G_c4_rssi_vs_distance.png}{0.85\textwidth}{Measured mean \mbox{RSSI} vs.\ distance from Campaign~C4 (scatter) with the theoretical \mbox{FSPL} model overlaid (line). Indoor multipath causes deviations from the \mbox{FSPL} model; document them as part of the distribution shift analysis in LH3-H.}{fig:lh3g-c4}

\subsection{Campaign C5: Beacon Count Scaling}

Campaign C5 is the most complex campaign because it involves multiple simultaneous transmitters and interleaved data. It is broken into five sub-configurations.

\subsubsection{C5 Sub-configurations}
\begin{table}[h!]
	\centering
	\renewcommand{\arraystretch}{1.3}
	\begin{tabularx}{\textwidth}{c c c c X}
		\toprule
		\textbf{Config} & \textbf{Beacons} & \textbf{Mode} & \textbf{Interval} & \textbf{Purpose} \\
		\midrule
		C5a & 1 & Fixed & 2\,s & Baseline (no multi-beacon interference) \\
		C5b & 2 & Fixed & 2\,s / 3\,s & Staggered (minimal collision probability) \\
		C5c & 2 & ALOHA & [1,3]\,s & Random access (some collisions expected) \\
		C5d & 3 & Fixed & 1\,s each (staggered) & High-rate, low-collision \\
		C5e & 3 & ALOHA & [0.5,2]\,s & Maximum contention \\
		\bottomrule
	\end{tabularx}
\end{table}

\subsubsection{Execution}
\begin{enumerate}[leftmargin=2em]
	\item For each sub-configuration, create the corresponding JSON config file. Use \texttt{num\_beacons} to record how many beacons were active. For multi-beacon runs, the \texttt{label\_beacon} field in the output CSV will come from the feature vector's \texttt{beacon\_id} field (since each row represents a single beacon's packet).
	\item Collect 300 packets per sub-configuration (note: for multi-beacon runs, 300 packets means 300 \emph{total} received frames, not 300 per beacon).
	\item For C5b through C5e, run the collection script for 5 minutes regardless of whether 300 packets are reached. Record the actual count; do not pad with a second run.
\end{enumerate}

\subsubsection{Validation}
\begin{enumerate}[leftmargin=2em]
	\item Verify that C5b shows a roughly 50/50 split of frames between Beacon~A and Beacon~B (the staggered fixed-interval configuration should deliver approximately equal numbers of frames from each beacon).
	\item Compare \PER{} between C5b (fixed, low collision) and C5c (ALOHA, higher collision). \mbox{ALOHA} should show increased \PER{} due to \RF{} collisions.
	\item For C5c and C5e, check the \texttt{inter\_arrival\_ms} distribution per beacon. Under \mbox{ALOHA}, inter-arrival times should vary more widely than under fixed scheduling.
	\item Watch for any \FIFO{} overflow alarms (\texttt{led(15)} on the Nexys A7) during C5d and C5e. If overflow occurs, record it in the campaign notes.
\end{enumerate}

\begin{tcolorbox}[infobox={What C5 Teaches Module~4}]
	The C5 dataset has two uses in Module~4:
	\begin{itemize}[leftmargin=1.2em]
		\item \textbf{Emitter identification:} the \texttt{label\_beacon} column provides ground truth for training a classifier that distinguishes Beacon~A from Beacon~B from Beacon~C based on their feature vectors. Even when all beacons transmit at the same power, SF, and modulation, subtle hardware differences (crystal frequency offset, RFM95W component tolerance) produce distinguishable \mbox{RSSI} and frequency-error signatures.
		\item \textbf{Scheduling mode classification:} the contrast between fixed-interval (C5b, C5d) and \mbox{ALOHA} (C5c, C5e) inter-arrival time distributions provides a basis for classifying a beacon's transmission scheduling policy --- a task with applications in spectrum monitoring.
	\end{itemize}
\end{tcolorbox}

\section{Verification Checklist}
\begin{itemize}[leftmargin=2em]
	\item[$\square$] C3: 600 rows, 300 per modulation type; \texttt{mod\_code} field correct for both runs.
	\item[$\square$] C4: 800 rows, 200 per distance; mean \mbox{RSSI} decreases with distance; \mbox{FSPL} overlay plot generated.
	\item[$\square$] C5: all five sub-configurations executed; C5b shows approximately equal beacon \mbox{ID} counts; C5c \PER{} $>$ C5b \PER{}.
	\item[$\square$] All CSV files use the same schema as C1/C2 (same column order and names).
	\item[$\square$] Any \FIFO{} overflow events during C5d/C5e documented in campaign notes.
\end{itemize}
