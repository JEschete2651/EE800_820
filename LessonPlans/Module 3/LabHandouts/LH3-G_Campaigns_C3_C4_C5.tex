\labheader{3-G}{Data Collection --- Campaigns C4 and C5}

\section{Objective}
Execute the two remaining single-variable sweep campaigns: physical distance (C4) and active-station count scaling (C5). These campaigns complete the controlled experimental dataset that drives all machine learning work in Module~4 and the operating-condition shift experiment.

\begin{tcolorbox}[infobox={Scope note: Campaign C3 removed}]
Campaign C3 was originally defined as a modulation-change sweep. It was removed from this handout because supporting that test at this stage would require a larger firmware and collection-pipeline rewrite than the schedule allows. The scope was trimmed to C4 and C5 to preserve a stable dataset contract for Module~4.
\end{tcolorbox}

\begin{tcolorbox}[warning={Preflight check before every campaign in this handout}]
The SX1276 receiver in the Threat is configured by a compile-time \texttt{LORA\_SF} define and cannot auto-detect spreading factors across LoRa configurations. If you finished LH3-F's C2 sweep with the Threat flashed at SF10 (or any value other than SF7), it will silently fail to demodulate Target~A's SF7 packets and \texttt{collect.py} will report \texttt{rx=0} indefinitely. Before starting C4 or C5:
\begin{itemize}[leftmargin=1.2em]
	\item Verify \texttt{LORA\_SF 7} and \texttt{LORA\_TX\_PWR\_DBM 14} in \texttt{Core/Src/main.c} of \emph{both} \texttt{Nucleo\_RFM95\_TargetA} and \texttt{Nucleo\_RFM95\_Threat}.
	\item For C5, also verify \texttt{LORA\_SF 7} and \texttt{LORA\_TX\_PWR\_DBM 14} in \texttt{Nucleo\_RFM95\_TargetB}.
	\item Press \textbf{F7} to build and \textbf{F5} to flash each board, then confirm each debug terminal prints \texttt{lora\_init done: SF=7 TxPwr=14 dBm} (or \texttt{TxPwr=20 dBm} on the Threat) before collecting.
\end{itemize}
If \texttt{collect.py} prints \texttt{[WARN] SF mismatch: reported $X$, config $Y$}, the LH3-F firmware fix to \texttt{lh1f\_packet.c} (\texttt{out[7] = g\_lora\_sf}) was reverted; restore it before continuing.
\end{tcolorbox}

\section{Procedure}

\subsection{Campaign C4: Physical Distance Sweep}

\subsubsection{Setup}
Configure Target~A: +14~dBm, SF7, LoRa, 2-second fixed initiator interval. Target~B off, Threat button not held (DATA-only collection). Vary only the physical separation between Target~A and the interceptor (Threat) antenna. Use a tape measure for all distance measurements and record to the nearest 0.5~m.

\begin{tcolorbox}[warning={Distance Measurement Protocol}]
	Measure distance from the \emph{antenna} of Target~A to the \emph{antenna} of the interceptor (Threat), not board-to-board. In an indoor environment, multipath reflections from walls, furniture, and the ground plane are unavoidable; the measured \mbox{RSSI} will deviate from the free-space path loss model. Record the measurement method and any significant obstructions (e.g., ``metal shelving at 2~m between Target~A and the interceptor'') in the campaign notes field. This information is required for the operating-condition shift analysis in LH3-H.
\end{tcolorbox}

\subsubsection{Execution}
\begin{enumerate}[leftmargin=2em]
	\item For each distance --- 1~m, 3~m, 5~m, 8~m --- place Target~A, confirm the Threat is receiving (check debug \UART{}), then run \texttt{collect.py} with the appropriate config JSON (\texttt{distance\_m} field must match). Collect 200 packets per distance.
	\item Between distance changes, verify Target~A is still transmitting (check its USART2 debug) and the Threat is still receiving before starting the next collection run.
\end{enumerate}

\subsubsection{Validation}
\begin{enumerate}[leftmargin=2em]
	\item Verify 800 total rows, 200 per distance, all with \texttt{pkt\_type = 0} (DATA).
	\item Compute mean \mbox{RSSI} per distance. It should decrease (become more negative) as distance increases.
	\item Overlay the theoretical free-space path loss (\mbox{FSPL}) curve. For the 906.5\,MHz carrier, TX power $P_{\text{tx}}$, and distance $d$:
	\[
		\text{\mbox{RSSI}}_{\text{theory}}(d) = P_{\text{tx}} + G_{\text{tx}} + G_{\text{rx}} - 20\log_{10}(d) - 20\log_{10}(906.5 \times 10^6) + 147.55
	\]
	where $G_{\text{tx}} = G_{\text{rx}} \approx 0$\,dBi for the spring antennas from LH1-A.
	\item Generate a scatter plot of measured mean \mbox{RSSI} vs.\ distance with the \mbox{FSPL} curve overlaid. Indoor measurements will typically show higher \mbox{RSSI} than \mbox{FSPL} at short distances (constructive multipath) and more variable behavior at longer distances.
\end{enumerate}

\subsection{Campaign C5: Active-Station Scaling}

Campaign C5 exercises the system with multiple simultaneous transmitters and interleaved DATA / ACK traffic. It is broken into two sub-configurations.

\begin{tcolorbox}[infobox={Collection script update: timed-run mode}]
\texttt{collect.py} was extended during this handout to support a timed collection mode required by C5b. If a campaign JSON contains a \texttt{collection\_duration\_s} field, the script runs for that many seconds and then stops automatically, regardless of whether \texttt{target\_packets} has been reached. Without the field, behaviour is unchanged (packet-count termination as in C1/C2). The C5b JSON (\texttt{campaign\_c5b.json}) sets \texttt{"collection\_duration\_s": 300}; no script flag or argument change is needed.
\end{tcolorbox}

\subsubsection{Execution}
\begin{enumerate}[leftmargin=2em]
	\item For each sub-configuration, use the pre-built JSON config file (\texttt{campaign\_c5a.json} or \texttt{campaign\_c5b.json}) in the \texttt{campaigns/} folder. The \texttt{num\_active\_stations} and \texttt{expected\_pkt\_types} fields are already set: \texttt{[0]} for C5a, \texttt{[0,1]} for C5b. The \texttt{label\_beacon} field in the output CSV comes from the feature vector's \texttt{beacon\_id} field (since each row represents a single station's packet).
	\item Collect 300 packets per sub-configuration (note: 300 packets means 300 \emph{total} received frames, summed across all active stations and all packet types).
	\item For C5b, \texttt{collect.py} will stop automatically after 5 minutes (the \texttt{collection\_duration\_s} field in the config). Record the actual packet count printed at exit; do not pad with a second run.
\end{enumerate}

\subsubsection{Validation}
\begin{enumerate}[leftmargin=2em]
	\item Verify that C5b shows a roughly 50/50 split of frames between Beacon~ID \texttt{0x01} (DATA) and Beacon~ID \texttt{0x02} (ACK) --- one ACK is generated per successful DATA, so the row counts should be close to equal under low collision.
	\item Watch for any \FIFO{} overflow alarms (\texttt{led(15)} on the Nexys A7) during C5b. If overflow occurs, record it in the campaign notes.
\end{enumerate}

\begin{tcolorbox}[infobox={What C5 Teaches Module~4}]
	The C5 dataset has two uses in Module~4:
	\begin{itemize}[leftmargin=1.2em]
		\item \textbf{Emitter identification:} the \texttt{label\_beacon} column provides ground truth for training a classifier that distinguishes Target~A vs.\ Target~B vs.\ Threat based on their feature vectors. Even when all stations transmit at the same power, SF, and modulation, subtle hardware differences (crystal frequency offset, RFM95W component tolerance) produce distinguishable \mbox{RSSI} and frequency-error signatures.
		\item \textbf{Operating-condition shift:} C5 (contended regime) vs.\ C1, C2, and C4 (clean regime) provides the canonical train/test split for the LH4-F operating-condition shift experiment. The accuracy drop from clean$\to$contended is the framework's central pedagogical observation about distribution shift, performed entirely on hardware data without any simulator dependency.
	\end{itemize}
\end{tcolorbox}
