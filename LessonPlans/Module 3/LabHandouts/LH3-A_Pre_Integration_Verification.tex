\labheader{3-A}{Pre-Integration Hardware Verification}

\section{Objective}
Verify that every subsystem produced in Modules~1 and~2 operates correctly \emph{in isolation} before any cross-board connections are made. This handout prevents the most common Module~3 mistake: connecting hardware before knowing which subsystem is at fault. By the end you will have a signed-off pre-integration checklist and a documented wiring plan that serves as the ground truth for debugging in Lab Handout~3-C.

\section{Prerequisites}
\begin{itemize}[leftmargin=2em]
	\item Lab Handout~1-G (interceptor / Threat firmware validated, forwarding frames confirmed on STM32 debug \UART{}, RSSI table populated, button-gated PAUSE TX confirmed).
	\item Lab Handout~1-F (cooperative-target firmware validated on both Target~A and Target~B, ping-pong sequence numbers and the \texttt{"I'm Jammed"} response confirmed).
	\item Lab Handout~2-H (Module~2 bitstream signed off, \texttt{ingest\_top.bit} generated and archived).
\end{itemize}

\section{Procedure}

\subsection{Step 1 --- Target~A verification}
\begin{enumerate}[leftmargin=2em]
	\item Flash Target~A with the LH1-F firmware compiled with \texttt{\#define ROLE\_INITIATOR} and \texttt{MY\_BEACON\_ID = BEACON\_ID\_A}. Power on via USB.
	\item Open a terminal emulator at the virtual COM port (115200\,8N1). Confirm that the debug \UART{} prints a CSV ping-pong line every $\sim$1 second:
	\begin{verbatim}
CSV,42,84000,84062,62,-32
	\end{verbatim}
	(\texttt{seq, t\_tx\_ms, t\_ack\_ms, rtt\_ms, rssi\_dbm}). Lines with \texttt{t\_ack\_ms} = \texttt{0} indicate Target~B is not yet responding --- that is expected at this point in Step~1 since Target~B has not been powered up yet.
	\item Verify the sequence number increments by 1 on each line. A skipped number indicates a missed transmit interrupt.
	\item Record the Beacon~ID (should be \texttt{0x01}) and the spreading factor.
	\item The Nucleo user \mbox{LED} (LD2, \texttt{PA5}) should not be required to blink for this validation --- LH1-F's TX-trigger pulse is on \texttt{PB10}, not LD2. If you have a scope, confirm the \texttt{PB10} pulse on each transmit cycle.
\end{enumerate}

\begin{tcolorbox}[warning={SPI Bring-Up Regression}]
	If no \texttt{CSV} lines appear in the terminal at all, the firmware has not started or the \RF{} hardware is not responding to the STM32 \mbox{SPI} init sequence. Re-flash with the Run and Debug \textbf{Start Debugging} arrow and confirm the \texttt{RegVersion = 0x12} and \texttt{lora\_init done} startup lines appear before the loop begins. Repeat LH1-C (\mbox{SPI} bring-up) if \texttt{RegVersion} returns anything other than \texttt{0x12}.
\end{tcolorbox}

\subsection{Step 2 --- Target~B verification and ping-pong validation}
\begin{enumerate}[leftmargin=2em]
	\item Flash Target~B with the LH1-F firmware compiled with \texttt{\#define ROLE\_RESPONDER} and \texttt{MY\_BEACON\_ID = BEACON\_ID\_B}. Power on via USB.
	\item Open a second terminal at Target~B's virtual COM port. Confirm \texttt{ACK seq=...} lines appear matched one-for-one to Target~A's outgoing DATA sequence numbers.
	\item Return to Target~A's terminal. The previously-zero \texttt{t\_ack\_ms} field should now be populated, with \texttt{rtt\_ms} typically $\sim$60--80\,ms at SF7 and \texttt{rssi\_dbm} between $-30$ and $-50$ at $\sim$1\,m.
	\item Place Target~A and Target~B within 1\,m of each other and confirm both run sustained ping-pong without locking up. Verify Target~A's reception rate (\texttt{t\_ack\_ms}~$\ne$~0 fraction) is $\geq$95\%.
\end{enumerate}

\subsection{Step 3 --- Interceptor (Threat) verification}
\begin{enumerate}[leftmargin=2em]
	\item Flash the third Nucleo with the Threat firmware from LH1-G (\texttt{Nucleo\_RFM95W\_Threat}). Power on via USB.
	\item Open a terminal at the Threat's virtual COM port (USART2, 115200\,8N1). With both Targets running ping-pong, confirm that the Threat's debug \UART{} prints a reception line for each captured packet:
	\begin{verbatim}
RX sender=01 type=0 rssi=-32 snr=9
RX sender=02 type=1 rssi=-31 snr=10
	\end{verbatim}
	(\texttt{type=0} = DATA from Target~A, \texttt{type=1} = ACK from Target~B). Confirm that both Beacon~IDs appear, sequence numbers are contiguous, and the \texttt{type} field correctly distinguishes DATA from ACK.
	\item Confirm that the Threat is also emitting 43-byte forwarding frames on \texttt{USART3 TX}. The easiest verification is to connect the Threat's \texttt{USART3 TX} pin to a USB-\UART{} adapter and capture raw hex on a third terminal session. The first byte should be \texttt{0x7E}.
	\item Check that \RSSI{} values are negative (reasonable for indoor distances) and that \mbox{SNR} is positive. Negative \mbox{SNR} at $\sim$1\,m distance indicates an antenna or front-end problem.
	\item \textbf{Validate the PAUSE TX path:} hold the Threat's user button (B1, blue button on the Nucleo) for 3\,s. The Threat's USART2 should print one \texttt{PAUSE -> ID=...} line every $\sim$100\,ms (about 30 lines total). The targeted Target's USART2 should print \texttt{"I'm Jammed"} once per pause-entry, and its scheduled DATA TX should pause for the duration of the button hold + 500\,ms hold-off.
\end{enumerate}

\subsection{Step 4 --- FPGA standalone verification}
\begin{enumerate}[leftmargin=2em]
	\item Connect the Nexys A7 to the host PC via USB. Vivado Hardware Manager should detect \texttt{xc7a100t\_0}.
	\item Program the \texttt{ingest\_top.bit} bitstream from LH2-H. The \mbox{DONE} \mbox{LED} should illuminate immediately after programming.
	\item Verify basic I/O: press the center pushbutton (reset). \texttt{led(0)} through \texttt{led(7)} should assert for one clock cycle after reset release (the Module~2 initialization pattern), then settle.
	\item Open a terminal at the Nexys A7 USB-\UART{} port (115200\,8N1). With no signal on the \mbox{PMOD} input, the pipeline should be idle --- no output lines should appear.
	\item Confirm the Vivado Hardware Manager can also run the \mbox{ILA} dashboard without errors (even if no triggers fire). This validates the \mbox{JTAG} path for Lab Handout~3-B.
\end{enumerate}

\begin{tcolorbox}[infobox={FPGA UART Loopback Test (Optional / Diagnostic)}]
	\textbf{Optional --- only run this check if your current \texttt{ingest\_top} bitstream explicitly wires a USB-\UART{} loopback path.} The canonical Module~2 build does not guarantee this path, so a non-echo result here does \emph{not} indicate a pipeline fault. If the loopback is present, verify by typing any character in the terminal and confirming it echoes back via the onboard FT2232H bridge. If the loopback is absent, skip this step --- it is not required for the interceptor forwarding path to work.
\end{tcolorbox}

Document the planned inter-board connections \emph{before} making them. Use the table below as a wiring record; fill in the actual pin numbers for your hardware revision.

	\end{tabularx}
\end{table}

\begin{tcolorbox}[warning={Do Not Connect Yet}]
	Fill in and photograph this wiring table \emph{before} making any connections. The purpose of this step is to think through the wiring plan while all subsystems are still running independently. Making connections prematurely is the leading cause of integration bugs that are actually wiring errors, not firmware or \RTL{} bugs.
\end{tcolorbox}

\section{Verification Checklist}
\begin{itemize}[leftmargin=2em]
	\item[$\square$] All 9 pre-integration checklist items pass.
	\item[$\square$] Debug \UART{} output verified for Target~A, Target~B, and the Threat simultaneously.
	\item[$\square$] Forwarding frame \texttt{0x7E} start delimiter confirmed from Threat \texttt{USART3 TX}.
	\item[$\square$] PAUSE protocol round-trip validated: button held on Threat $\to$ \texttt{"I'm Jammed"} on targeted Target within $\sim$100\,ms.
	\item[$\square$] \FPGA{} bitstream programs cleanly; Vivado \mbox{ILA} dashboard accessible.
	\item[$\square$] Wiring plan documented before any connections made.
\end{itemize}

\begin{tcolorbox}[deliverable={Lab Handout 3-A Deliverable}]
	Submit the following before beginning Lab Handout~3-B:
	\begin{enumerate}[leftmargin=1.2em]
		\item Signed-off pre-integration checklist (all 9 items checked).
		\item Photos of the physical setup: each Nucleo board powered and running, Nexys A7 programmed.
		\item Terminal capture screenshots: Target~A CSV output (at least 5 lines), Target~B ACK output (at least 5 matched lines), Threat debug output showing interleaved DATA/ACK rows plus a button-hold burst of PAUSE lines.
		\item Hex capture of at least one complete 43-byte forwarding frame from the Threat \texttt{USART3 TX} (annotated to identify the \texttt{0x7E} start byte, length field, Beacon~ID at frame offset~12, and the packed \texttt{Modulation Code} byte at frame offset~16 with bit-field decoding).
		\item Completed wiring table (photographed or typed).
	\end{enumerate}
\end{tcolorbox}
